# Jerry-Insight Agent 面试准备稿

## 1. 项目定位

省钱智探 Agent 是一个面向个人消费决策的 Agent 应用系统，不是单轮问答机器人。

它解决的问题是：用户在消费决策中经常遇到“买前查价难、买后记账散、历史消费难复盘”的问题。系统把商品问价、实时比价、RAG 历史记忆、预算扣减、钉钉通知、直接记账、余额加回、撤销流水和历史持久化串成闭环。

推荐面试开场：

```text
这个项目是一个消费风控 Agent。我把它设计成固定业务流程下的 Agent 应用，而不是完全自由规划的聊天机器人。系统会先做输入分流，区分商品问价、直接记账、余额加回、撤销和普通问答；如果是商品问价，就进入 FSM 流程，依次完成意图识别、实体清洗、RAG 检索、Web Search/价格爬虫、LLM 审计、账本确认和钉钉通知。
```

```text
省钱智探 Agent：消费风控与记账智能体｜个人项目｜2026.02 - 2026.05

概述：面向个人消费决策中“买前查价难、买后记账散、历史消费难复盘”的问题，基于 Python / Streamlit 独立开发消费风控 Agent，支持商品问价、实时比价、RAG 历史记忆、预算扣减、钉钉通知、直接记账、余额加回、撤销流水与历史持久化，形成“问价-审计-扣款-通知-复盘”的端到端闭环。

● Agent 流程控制：自研 FSM + Harness 执行层，将消费审计拆分为意图识别、实体清洗、RAG 检索、价格搜索、审计生成和账本确认等状态节点，显式控制工具调用、异常退出与结果落库，降低黑盒 Agent 执行流不可控风险。

● RAG 混合检索：基于 ChromaDB 记录历史购买、拦截和偏好信息，结合 Jaccard 相似度与重排序策略，将用户历史记忆、实时搜索结果和价格爬虫信息融合进审计上下文，提升口语化商品查询下的个性化判断能力。

● 异步调度与持久化：使用 ThreadPoolExecutor 并行执行 RAG 检索、Web Search、价格爬虫和钉钉通知，避免外部 I/O 阻塞前端；在模拟 10 个外部通知任务测试中，主线程入队耗时约 1.47ms；使用 JSON / SQLite 持久化用户画像、历史会话和消费流水，并通过 threading.Lock 保护账本写入。

● 稳定性与可观测：针对模型输出格式异常、工具调用超时、云端 Secrets 缺失、前端状态锁死等问题，设计格式解析兜底、默认价格修正、状态自动释放和异步通知失败隔离机制；构建 Agent Trace 日志记录各阶段耗时与异常状态，为性能优化提供数据支撑。
```

## 3. 这是不是 Toy 项目？

推荐回答：

```text
它不是企业级 Agent 平台，但我把它当作一个完整 Agent 应用系统来做。核心不在规模，而在于把 Agent 工程里常见的问题落到具体业务里：输入意图混杂、工具调用不稳定、LLM 输出格式异常、外部 I/O 阻塞、历史记忆缺失、账本状态持久化和云端配置隔离。项目里这些问题都有对应处理，也补了 trace 和回归测试。
```

可以强调：

- 它不是单纯调一次 LLM。
- 它有工具调用：RAG、Web Search、价格爬虫、账本、钉钉。
- 它有状态控制：FSM + Harness。
- 它有记忆：ChromaDB + 历史记录。
- 它有业务闭环：问价 -> 审计 -> 扣款/放弃 -> 通知 -> 历史复盘。
- 它有稳定性处理：格式解析、超时降级、状态释放、异步通知。
- 它有评测：输入分流、账本操作、异步隔离、Trace。

## 4. 为什么不用 LangChain / AgentExecutor？

推荐回答：

```text
我的任务链路比较固定，不是开放式自由探索任务。核心流程是意图识别、实体清洗、RAG 检索、搜索/爬虫、审计生成、账本确认和通知。如果完全交给黑盒 AgentExecutor，模型可能循环调用工具、输出格式不稳定，状态恢复也不好做。所以我选择 FSM + Harness，把系统调度权放在代码层，LLM 主要负责审计判断和报告生成。
```

补充：

- AgentExecutor 更适合工具选择自由度较高的任务。
- 你的项目更关注稳定业务流程，所以显式状态机更合适。
- 这不是否定 LangChain，而是根据场景做技术取舍。

## 5. FSM / Harness 怎么讲？

FSM 负责大流程状态：

```text
INTENT_CHECK -> PRICE_SCOUT -> AUDIT_REPORT -> END
```

Harness 负责 LLM 审计阶段的工具调用和收敛：

- 要求 LLM 输出结构化 JSON。
- 允许触发 Web Search。
- 超过步数或格式异常时进入强制收敛。
- 最终必须输出包含 PRICE_DATA 的审计结果。

推荐回答：

```text
FSM 负责控制业务流程，Harness 负责约束 LLM 在审计阶段的推理和工具调用。这样可以把“流程稳定性”和“模型判断能力”分开：代码控制状态，模型负责审计文本和价格判断。
```

## 6. RAG 在项目里做了什么？

不是简单知识库。它存的是用户消费历史和偏好：

- 买过什么。
- 拦截过什么。
- 用户偏好和雷点。
- 历史审计结果。

推荐回答：

```text
RAG 在这个项目里不是通用问答知识库，而是消费记忆层。它记录用户历史购买、拦截和偏好信息，在后续商品审计时作为上下文输入。比如用户之前放弃过某类商品，后续再问类似商品时，系统可以把历史拦截记录作为审计依据。
```

RAG 链路：

```text
历史记录/偏好 -> ChromaDB 存储 -> 检索 -> Jaccard / rerank -> 拼入审计上下文 -> LLM 生成建议
```

可能被问：

```text
为什么用 ChromaDB？
```

回答：

```text
项目规模不大，ChromaDB 本地部署简单，适合个人项目快速做向量记忆和本地持久化。后续如果数据规模扩大，可以迁移到 Milvus、Qdrant 或云数据库。
```

## 7. 输入治理怎么讲？

你的输入分流覆盖：

- 普通帮助：怎么用、你能干嘛。
- 商品问价：我想买可乐、iPhone 值不值得买。
- 直接记账：买了雪碧花了 3 块。
- 余额加回：加回来 3 块。
- 撤销：撤销上一笔、加错了。

推荐回答：

```text
真实用户输入不会都像标准命令，所以我做了规则优先的输入治理。先用确定性规则拦截直接记账、余额加回、撤销和帮助类问题，避免这些简单动作进入 LLM 审计链路。只有商品问价才进入完整 Agent 审计流程。这样能减少误触发，也能降低调用成本和状态复杂度。
```

已有数据：

```text
基于 38 条自建消费场景回归样本，意图分流、金额提取和账本操作均通过测试。
```

注意说法：

- 简历写“均通过测试”比直接吹“工业级 100%”更稳。
- 如果被问样本多少，如实说 38 条自建样本，覆盖主要输入类型。

## 8. 异步调度怎么讲？

异步对象：

- RAG 检索。
- Web Search。
- 价格爬虫。
- 钉钉通知。

推荐回答：

```text
这些任务都属于外部 I/O 或耗时操作，如果串行执行，会拖慢 Streamlit 前端响应。我使用 ThreadPoolExecutor 并行调度检索、搜索、爬虫和通知，把网络延迟从主链路中隔离出来。钉钉通知即使失败，也不会影响审计结果展示和账本操作。
```

已有数据：

```text
模拟 10 个外部通知任务，主线程入队耗时约 1.47ms。
```

注意：

- 不要说整个 Agent 只耗时 1.47ms。
- 这是“异步任务入队耗时”，不是完整审计耗时。

## 9. 稳定性兜底怎么讲？

你项目里真实发生过的坑：

- LLM 输出格式异常。
- PRICE_DATA 解析失败。
- 工具调用超时。
- 钉钉通知被前置异常挡住。
- 直接记账后输入框锁死。
- 本地没有 secrets.toml 导致无法启动。
- 历史只在 session，刷新后丢失。

对应解决：

- PRICE_DATA 正则/JSON 双路径解析。
- 默认价格兜底和饮料价格纠偏。
- 爬虫 timeout 降级。
- 通知与日志保存解耦。
- SUBMIT_PROCESSING 状态自动释放。
- safe_secret_get 兼容本地 env 和云端 secrets。
- 历史、账本、trace 写入 JSON/JSONL。

推荐回答：

```text
Agent 应用里最容易出问题的是外部工具和模型输出不稳定。所以我做了几类兜底：输出格式解析兜底、工具超时降级、通知失败隔离、前端状态自动释放，以及本地/云端 secrets 兼容。目标不是保证每个工具都成功，而是保证单点失败不阻塞主流程。
```

## 11. 本地版和云端版差异

本地版：

- 使用 `.env`。
- 使用 `tools.search_local`。
- 使用本地 Chroma `memory_bank`。
- 更适合本地调试和数据保存。

云端版：

- 使用 Streamlit Secrets。
- 使用 `tools.search`。
- 云端 Chroma 可能是临时的 EphemeralClient。
- 更适合演示和线上访问。

推荐回答：

```text
本地版主要用于调试和持久化验证，云端版用于线上演示。两者核心业务能力已尽量对齐，但云端环境文件持久化不如本地稳定，所以我把 API Key 和 Webhook 放到 Streamlit Secrets，并把数据目录和持久化边界作为后续优化方向。
```

## 14. 高频面试问题与回答

### Q1：你的项目为什么算 Agent？

```text
它不是单轮 LLM 问答，而是一个有工具调用和状态流转的应用。系统会根据用户输入分流，如果是商品问价，就触发 RAG 检索、Web Search、价格爬虫和 LLM 审计；如果用户确认购买，还会调用账本工具完成扣款、通知和历史归档。LLM 负责审计判断，FSM/Harness 负责流程控制和兜底。
```

### Q2：为什么不用纯 LangChain AgentExecutor？

```text
因为这个项目是固定业务流程，不是开放式任务规划。纯 AgentExecutor 自由度高，容易工具循环调用、输出格式不稳定，状态也不好恢复。我用 FSM 显式控制主流程，用 Harness 约束 LLM 审计阶段，这样稳定性更好。
```

### Q3：RAG 检索到的东西怎么影响回答？

```text
RAG 检索到的是用户历史购买、拦截和偏好信息，会拼接进 LLM 审计上下文。LLM 在生成消费建议时，会同时参考用户余额、历史偏好、实时搜索结果和价格爬虫信息。
```

### Q4：工具调用失败怎么办？

```text
不同工具有不同兜底。价格爬虫超时就降级使用 Web Search 数据；RAG 失败就返回“历史档案隔离状态”；钉钉通知失败不会阻塞主链路；LLM 输出格式异常则通过 JSON/正则解析和默认价格兜底处理。
```

### Q5：怎么证明项目有效？

```text
我做了两类评测。第一类是输入分流和账本操作回归测试，38 条自建消费场景样本覆盖帮助问答、商品问价、直接记账、余额加回、撤销等类型，分流、金额提取和账本操作均通过测试。第二类是异步隔离测试，模拟 10 个外部通知任务，主线程入队耗时约 1.47ms。后续我还补了 Agent Trace，用于统计每个审计阶段耗时。
```

### Q6：项目最大的技术难点是什么？

```text
不是单个算法，而是 Agent 工程稳定性。真实输入很混杂，工具调用可能超时，LLM 输出格式不稳定，前端状态也可能锁死。所以我主要做了三件事：第一，用 FSM 控制流程；第二，用规则分流把确定性动作前置；第三，给工具调用、通知、输出解析和状态恢复加兜底。
```

### Q7：如果继续优化，你会做什么？

```text
第一，完善 Agent Trace 评测，统计端到端耗时和各阶段瓶颈。第二，做价格准确性评测，构建商品参考价数据集，统计误差。第三，把云端 JSON 持久化迁移到外部数据库，比如 Supabase 或 Postgres，提升云端数据可靠性。
```

## 16. 一分钟项目介绍

```text
我做的是一个个人消费风控 Agent，目标是解决买前查价、买后记账和历史消费复盘的问题。系统会先做输入分流，区分商品问价、直接记账、余额加回、撤销和普通问答。对于商品问价，会进入 FSM + Harness 控制的审计流程，依次完成实体清洗、RAG 历史检索、Web Search、价格爬虫和 LLM 审计；用户确认后会扣减预算、写入账本、推送钉钉并保存历史。

技术上我没有完全依赖黑盒 AgentExecutor，而是用 FSM 显式控制流程，用 Harness 约束 LLM 的工具调用和输出格式。项目里还处理了工具超时、输出格式异常、通知失败、输入框状态锁死、本地和云端 secrets 差异等问题。为了证明效果，我补了 38 条输入分流回归测试和 Agent Trace 日志，用来统计意图分流、账本操作和各阶段耗时。
```
