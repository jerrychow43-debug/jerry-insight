## 3. 项目介绍标准版

我做的是一个消费决策 Agent。用户可以输入“我想买可乐”“我想买耳机”这类商品问题，系统会先做意图识别和商品实体提取，然后进入一个 FSM / Harness 控制的执行链路。

执行链路里会并行调用几个能力模块：RAG 检索历史消费和对话记忆，Web Search 获取全网情报，价格爬虫抓取什么值得买等价格线索，最后交给 LLM 做消费审计，输出是否建议购买、理由、估算单价和可追溯来源。

项目还支持直接记账、确认购入扣款、退款加回、撤销上一笔、钉钉通知和历史记录。为了方便评估，我加入了 Trace 日志，记录每次 Agent 在意图识别、实体抽取、RAG、搜索、爬虫、LLM 审计等阶段的耗时和状态，也写了基础回归测试脚本验证意图分流、金额提取和账本操作。

## 5. 核心概念回答

### 你怎么理解 Agent？

Agent 不是简单的大模型聊天，而是 LLM 加上工具、状态、记忆和执行流程。LLM 负责理解和决策，工具负责外部能力，比如搜索、数据库、通知；状态负责记录当前任务进展和用户数据；评测负责判断 Agent 是否稳定可靠。

在我的项目里，LLM 不是直接处理所有事情。商品问价会进入 Agent 链路，但直接记账、撤销这种确定性任务走规则和数据库，这样稳定性更高。

### Prompt 和 Skill 的区别？

Prompt 更像是给模型的角色、任务和输出约束。Skill 是 Agent 可以调用的能力模块，比如搜索、爬虫、RAG、数据库写入、发通知。

Prompt 决定 Agent 怎么思考和怎么输出，Skill 决定 Agent 能实际做什么。

### RAG 是什么？

RAG 是 Retrieval-Augmented Generation，检索增强生成。它不是只靠模型参数回答，而是在生成前先从外部知识库或历史记录里检索相关内容，再把这些内容作为上下文交给模型。

我的项目里 RAG 主要用于检索用户历史消费、对话记录和账本上下文，让购买建议不只依赖当前一句话。

### FSM / Harness 是什么？

FSM 是有限状态机，用来控制 Agent 的执行阶段，比如意图识别、商品提取、价格搜索、审计报告、结束。

Harness 是执行控制器，用来包住 LLM 和工具调用过程，限制最大步骤，处理工具调用失败，最后强制收敛成可解析结果。

通俗说，FSM 管“流程走到哪一步”，Harness 管“这一步里 LLM 怎么调用工具、怎么输出结果”。

### 为什么要做意图分流？

因为不是所有输入都适合交给 LLM。比如“买了雪碧花了3块”应该直接记账，不需要大模型推理；“撤销上一条”应该直接操作数据库。如果全部交给 LLM，会增加延迟、成本和不稳定性。

所以我把普通帮助、商品问价、直接记账、退款加回、撤销拆成不同路径。这样确定性任务更稳定，复杂商品判断再交给 Agent。

### 工具调用失败怎么办？

我会做三层处理：

第一，设置超时，避免搜索或通知一直卡住。

第二，失败后降级，比如搜索失败时仍然基于历史记忆、爬虫线索和模型常识给保守建议。

第三，记录 Trace 和错误信息，方便后面定位是哪个阶段失败。

### 怎么评测 Agent？

我会从几个维度评测：

- 意图识别准确率：输入是否分到正确业务路径。
- 实体抽取准确率：商品名是否提取正确。
- 金额提取准确率：记账金额是否正确。
- 工具调用成功率：搜索、爬虫、通知是否成功。
- 延迟：总耗时和每阶段耗时。
- 输出可解析率：是否包含结构化价格数据。
- 用户验收：建议是否有来源、是否符合消费场景。

## 6. 可能被问到的问题和回答

### 你的项目最有技术含量的地方是什么？

不是单纯调用 LLM，而是把 Agent 拆成了可控链路：意图分流、工具调用、状态管理、RAG 记忆、Trace 日志和回归测试。这样系统不是一次性 Prompt Demo，而是可以定位问题、可以迭代的工程原型。

### 你的项目有什么不足？

目前主要有三个不足。

第一，评测集规模还不够大，更多是自建样本，后续需要扩充真实用户场景。

第二，价格准确性依赖搜索和爬虫质量，外部网站结构变化会影响结果。

第三，云端版本目前用 SQLite 做演示可以，但长期多用户访问应该迁移到云数据库。

### 如果让你给 IM 产品配置一个 Agent，你会怎么做？

我会先明确 Agent 的角色和边界，比如它是客服、日程助手、销售助手还是知识库助手。然后拆 Skills，比如消息理解、知识库检索、外部 API 调用、任务创建、通知发送。接着设计 Prompt 和输出格式，限制工具调用条件和失败降级方式。最后做测试集，覆盖正常问题、边界问题、无关问题、工具失败和多轮上下文。

### 你对 Prompt Engineering 怎么理解？

我理解 Prompt Engineering 不只是写一段好看的提示词，而是把角色、任务边界、输入上下文、工具使用条件、输出格式和失败处理都设计清楚。特别是工程里，输出格式很重要，否则后端很难解析和落库。

### 你怎么保证 Agent 不乱跑？

我会做几个限制：

- 意图分流，不相关输入不进入复杂链路。
- 工具调用有最大步数和超时。
- Prompt 明确输出 JSON 或固定结构。
- 后端做二次解析和兜底。
- Trace 记录每一步，发现异常再优化。

### 为什么从 Streamlit 迁移到 Vue + FastAPI？

Streamlit 适合快速验证原型，但前后端耦合比较强。为了更接近真实工程，我把它迁移成 Vue + FastAPI。前端负责交互展示，后端负责 Agent 逻辑、数据库、通知和 API。这样更符合前后端分离，也方便部署和扩展。

### 你用过哪些 AI 工具？

我长期使用 ChatGPT 和 Codex 辅助项目开发，包括需求拆解、代码修改、问题排查、README 和面试材料整理。但我不是直接复制结果，会结合运行日志和测试结果判断是否可用。比如这个项目里很多 bug 都是通过日志定位，然后逐步回退、修复和验证的。
