============================================================
一、项目一句话
============================================================

省钱智探 Agent 是一个面向个人消费决策和记账场景的 Agent 应用，围绕“买前问价、买后记账、历史复盘”构建了商品审计、RAG 历史记忆、账本操作、钉钉通知和 Trace 可观测闭环。

面试开场可以这样说：

这是一个个人消费风控与记账 Agent。我没有把它做成泛聊天机器人，而是围绕固定业务场景做流程控制。用户输入后先做意图分流，区分商品问价、直接记账、余额加回、撤销和普通问答；商品问价会进入 FSM / Harness 审计链路，依次完成实体抽取、RAG 历史检索、Web Search、价格信息处理、LLM 审计、账本确认和钉钉通知。

============================================================
二、一分钟项目介绍
============================================================

我做的是一个个人消费风控 Agent，主要解决买前查价难、买后记账散、历史消费难复盘的问题。系统会先对用户输入做规则优先的分流，如果是直接记账、余额加回或撤销，就走确定性账本链路；如果是商品问价，就进入 FSM + Harness 控制的审计流程。

在商品审计链路中，系统会抽取商品实体，检索 ChromaDB 中的历史购买和拦截记录，再结合 Tavily Web Search、价格线索和用户当前余额，交给 LLM 生成消费建议。用户确认后，系统会写入账本、扣减余额、保存历史并异步推送钉钉通知。

后期我补了 Agent Trace，记录意图识别、实体抽取、RAG、Web Search、LLM 审计等阶段耗时和异常状态。基于 Trace 我定位到 Web Search 是主要瓶颈，把深度搜索和二次网页抓取改成轻量搜索并加入超时降级，10 条复测样本中平均链路耗时从约 51 秒降到约 8.75 秒。

============================================================
三、技术栈怎么说
============================================================

技术栈：
Agent FSM / Harness、RAG、ChromaDB、Jaccard Rerank、Tavily Web Search、DeepSeek API、ThreadPoolExecutor、Streamlit、SQLite / JSON、DingTalk Webhook

不要主动写：
LangChain、LangGraph、标准 MCP、FastAPI

原因：
这些不是当前主云端版真正使用的核心技术，硬写容易被追问。

============================================================
四、项目主流程
============================================================

用户输入
-> 输入路由
   -> 普通问答 / 使用说明
   -> 直接记账
   -> 余额加回
   -> 撤销上一笔
   -> 商品问价审计
      -> 意图识别
      -> 商品实体抽取
      -> RAG 历史检索
      -> Web Search / 价格线索
      -> LLM 审计报告
      -> 用户确认扣款或放弃
      -> 账本持久化 + 钉钉通知 + 历史归档

重点说法：
我把项目拆成两类链路。确定性强的账本操作，比如直接记账、加回余额、撤销上一笔，优先用规则处理；不确定性强的商品问价才进入完整 Agent 审计链路。这样可以降低 LLM 误触发，也能减少不必要的调用。

============================================================
五、为什么这个项目算 Agent
============================================================

不要只说“用了大模型”。

要说：
它不是单轮 LLM 问答，而是一个有状态流转和工具调用的应用。商品问价会触发 RAG 检索、Web Search、价格信息处理和 LLM 审计；用户确认后还会写账本、扣余额、发钉钉和保存历史。LLM 负责消费审计判断，FSM / Harness 负责流程控制、工具调用约束和异常收敛。

可以强调：
1. 有输入路由，不是所有请求都直接进 LLM。
2. 有工具调用，包括 RAG、Web Search、账本、通知。
3. 有历史记忆，基于 ChromaDB 和本地历史数据。
4. 有状态控制，使用 FSM / Harness。
5. 有业务闭环，问价后可以确认扣款、放弃购买、历史归档。
6. 有可观测性，Trace 记录各阶段耗时和异常。

============================================================
六、FSM / Harness 怎么解释
============================================================

FSM 负责主业务流程：
INTENT_CHECK -> PRICE_SCOUT -> AUDIT_REPORT -> END

Harness 负责 LLM 审计阶段：
1. 约束 LLM 输出 JSON。
2. 根据模型输出判断是否需要二次搜索。
3. 控制最大推理步数。
4. 格式异常时做强制收敛。
5. 最终要求输出审计建议和 PRICE_DATA。

推荐回答：
FSM 更偏业务流程控制，Harness 更偏 LLM 执行约束。这样可以把流程稳定性和模型判断能力分开：状态流转由代码控制，审计理由和价格判断由 LLM 生成。相比完全黑盒 AgentExecutor，这种方式更容易插入 Trace、超时降级、账本确认和状态恢复。

============================================================
七、为什么不用 LangChain / LangGraph
============================================================

推荐回答：
这个项目不是开放式任务规划，而是比较固定的消费审计流程。核心链路是输入分流、实体抽取、RAG、搜索、审计、账本确认和通知。如果完全交给黑盒 AgentExecutor，工具循环、格式异常和状态恢复会更难控制。所以我选择用 FSM / Harness 自己控制流程，LLM 主要负责审计判断和报告生成。

如果被问 LangGraph：
LangGraph 更适合复杂多节点图和多 Agent 协作。如果后续要扩展成多个角色，比如价格分析 Agent、预算规划 Agent、账本 Agent，可以考虑迁移到 LangGraph。目前这个项目链路比较固定，用自研 FSM 更轻量，也更方便做前端状态和 Trace。

============================================================
八、RAG 在这里做什么
============================================================

RAG 不是通用知识库，而是消费记忆层。

保存的信息：
1. 历史购买记录。
2. 历史放弃购买记录。
3. 用户偏好和雷点。
4. 历史审计上下文。

推荐回答：
RAG 在这个项目里主要解决个性化上下文问题。系统会记录用户历史购买、拦截和偏好信息，后续用户再问类似商品时，会把相关历史作为审计上下文输入给 LLM。比如用户之前多次买过饮料，系统就能结合历史消费习惯给出更贴近个人情况的建议。

RAG 链路：
历史消费/偏好 -> ChromaDB -> 检索召回 -> Jaccard 重排 -> 拼入审计上下文 -> LLM 审计

为什么用 ChromaDB：
ChromaDB 本地部署简单，适合个人项目快速实现向量记忆和本地持久化。当前数据规模不大，ChromaDB 足够用。后续如果用户量或数据规模扩大，可以迁移到 Qdrant、Milvus、PostgreSQL + pgvector 或云向量库。

============================================================
九、输入治理与账本闭环
============================================================

支持的输入类型：
1. 帮助问答：怎么用、你能干嘛。
2. 商品问价：我想买可乐、MacBook 值不值得买。
3. 直接记账：买了雪碧花了 3 块。
4. 余额加回：加回来 3 块、退回 10 元。
5. 撤销上一笔：撤销上一笔、加错了、上一条记错了。

推荐回答：
真实用户输入不会都像标准命令，所以我做了规则优先的输入治理。直接记账、余额加回、撤销和帮助类问题都可以用确定性规则处理，只有商品问价才进入完整 Agent 审计流程。这能减少误触发，也能降低 LLM 调用成本。

已有数据：
基于 38 条自建消费场景回归样本，意图分流、金额提取和账本操作均通过测试。

注意：
可以说“38 条自建场景样本均通过”。
不要说“工业级准确率 100%”。

============================================================
十、异步通知与持久化
============================================================

为什么用 ThreadPoolExecutor：
Streamlit 前端对阻塞比较敏感，外部 Webhook 和网络搜索如果同步执行，容易导致页面卡顿。所以我用 ThreadPoolExecutor 把通知和部分外部 I/O 异步化。钉钉通知失败不会影响主链路，账本写入和前端展示也不会因为 Webhook 延迟被卡住。

已有数据：
模拟 10 个外部通知任务，主线程入队耗时约 1.47ms。

注意：
这是“通知任务入队耗时”，不是完整 Agent 响应耗时。

持久化文件：
jerry_profile.json：用户预算和余额。
jerry_history_sessions.json：历史会话。
jerry_ledger.json：账本流水。
jerry_trace_logs.jsonl：Trace 日志。

============================================================
十一、Trace 可观测与性能优化
============================================================

Trace 记录阶段：
intent_check
entity_extract
rag_retrieve
web_search
price_crawler
llm_audit

推荐回答：
我给 Agent 链路补了阶段级 Trace，每次商品审计都会记录各阶段耗时、状态和异常。这样能知道慢在哪里，而不是只知道“整体慢”。比如我通过 Trace 发现原始版本主要瓶颈在 Web Search。

优化前后数据：
优化前：16 条真实商品问价样本，平均总耗时约 51s，Web Search 平均约 32s。
优化后：10 条复测样本，闭环成功率 100%，异常数 0，timeout 0，平均总耗时约 8.75s。
降幅：约 82.85%。

优化做了什么：
1. 将 Tavily advanced 搜索改为 basic 搜索。
2. 减少搜索结果数量。
3. 去掉慢的二次网页正文抓取。
4. 加入 25 秒超时降级。
5. Trace 面板支持查看阶段耗时和下载 CSV。

简历和面试可说：
基于 Trace 定位 Web Search 为主要性能瓶颈，将深度搜索与二次网页抓取调整为轻量检索并加入超时降级；10 条复测样本中闭环成功率 100%、异常数 0，平均链路耗时由约 51s 降至约 8.75s。

============================================================
十二、Web Search 为什么会慢
============================================================

如果被问为什么之前慢：
原始版本搜索链路比较重：先做 Tavily 深度搜索，再对多个网页结果做二次正文抓取。这在云端环境会受外部网络和网页响应影响，长尾延迟很明显。

如果被问轻量搜索会不会影响质量：
会有一定取舍。所以我没有直接取消搜索，而是改成使用 Tavily 返回的摘要和来源链接，减少二次网页抓取；同时保留 RAG、价格线索和 LLM 审计。这样牺牲一部分深度网页正文信息，换取更稳定的响应时间。后续可以通过缓存和价格评估集继续平衡质量与延迟。

============================================================
十三、稳定性兜底
============================================================

处理过的问题：
1. LLM 输出格式异常。
2. PRICE_DATA 解析失败。
3. Web Search 超时。
4. 爬虫超时。
5. 钉钉通知失败。
6. Streamlit 输入框状态锁死。
7. 云端缺 Secrets。
8. 历史刷新后丢失。

对应机制：
1. JSON / 正则双路径解析价格。
2. 饮料等低价商品默认价格纠偏。
3. Web Search 25 秒超时降级。
4. 爬虫超时后不阻塞主链路。
5. 钉钉异步通知，失败不影响主链路。
6. SUBMIT_PROCESSING 状态自动释放。
7. safe_secret_get 同时兼容本地 env 和 Streamlit Secrets。
8. 历史、账本、Trace 写入本地文件。

推荐回答：
Agent 应用里最大的不稳定来源是外部工具和模型输出。所以我做的不是保证每个工具都一定成功，而是让单点失败不要拖垮主流程。比如 Web Search 超时就记录到 Trace 并降级继续，钉钉通知失败也不会影响账本写入和结果展示。

============================================================
十四、本地版和云端版差异
============================================================

本地版：
1. 更适合调试。
2. 使用本地 .env。
3. 保留本地搜索和本地记忆逻辑。
4. 曾实现 JSON-RPC 风格工具网关封装记账工具。

云端版：
1. 面向 Streamlit Cloud 演示。
2. 使用 Streamlit Secrets。
3. 更关注稳定性和前端可用。
4. 主记账链路采用直接函数调用。

推荐回答：
本地版主要用于调试和验证，云端版主要用于线上演示。两者核心功能尽量对齐，但云端环境对文件持久化和 Secrets 管理更敏感，所以云端版更强调稳定性和直接链路。比如本地版做过 JSON-RPC 风格工具网关，云端版为了降低部署复杂度，主记账链路采用直接函数调用。

============================================================
十五、MCP 被问到怎么说
============================================================

如果面试官问 MCP：
MCP 是 Model Context Protocol，用于让 Agent 通过标准协议调用外部工具和上下文资源。我这个项目没有完整接入标准 MCP 生态，但本地版实现过一个类 MCP / JSON-RPC 风格的工具网关，支持 tools/list 和 tools/call，并把 record_expense 封装成记账工具。云端版后来为了部署稳定性，把主链路改成了直接函数调用。

不要主动说：
标准 MCP 协议完整接入。

更稳说法：
JSON-RPC 风格工具网关 / 类 MCP 工具调用网关。

十六、价格准确性怎么说
============================================================

目前可以说：
项目已经支持价格线索检索和审计估价，但价格准确率还没有做大规模评估。当前更完整的数据是链路成功率、Trace 耗时和输入分流回归测试。后续可以构建价格参考集，统计估价误差、命中容忍区间比例和异常样本。

不要说：
1. 价格准确率 90%。
2. 全网最低价。

更稳说法：

============================================================
十七、高频面试题
============================================================

Q1：你这个项目最大的技术难点是什么？
A：最大难点不是单个算法，而是 Agent 工程稳定性。真实输入很混杂，外部工具可能超时，LLM 输出格式不稳定，Streamlit 前端也可能因为状态没释放而锁死。所以我主要做了输入分流、FSM 流程控制、工具超时降级、异步通知隔离和 Trace 可观测。

Q2：怎么证明它不是简单套壳？
A：如果只是套壳，就是把用户输入直接发给 LLM。但这个项目有明确的业务状态和工具链路：输入先分流，商品问价会触发 RAG、Web Search、价格信息处理和审计生成；确认购买后会写账本、扣余额、通知和归档。并且我补了回归测试和 Trace 数据，能看到每个阶段的耗时和异常。

Q3：为什么不用所有请求都走 LLM？
A：因为直接记账、加回余额、撤销上一笔这类操作确定性很强，用规则更稳定、成本更低，也更容易保证账本正确。LLM 更适合处理商品审计这种需要综合判断的部分。

Q4：RAG 检索不到怎么办？
A：检索不到就返回“暂无历史档案”，不会阻塞主链路。审计会继续基于当前输入、Web Search 和价格线索生成。RAG 是增强上下文，不是唯一依赖。

Q5：Web Search 超时怎么办？
A：设置了 25 秒超时。超时后会在 Trace 中记录 timeout，并基于已有历史、价格线索和模型常识继续生成审计建议。目标是保证外部工具失败不拖垮主流程。

Q6：钉钉通知失败怎么办？
A：钉钉通知是异步任务，失败不会影响审计结果展示和账本写入。这样把外部 Webhook 从主业务链路中隔离出来。

Q7：账本写入怎么保证一致性？
A：目前是个人项目，使用 JSON / SQLite 做本地持久化，并用 threading.Lock 保护文件写入，避免同一进程内并发写账导致数据错乱。后续如果做多用户或多实例部署，需要迁移到数据库事务，比如 PostgreSQL 或 Supabase。

Q8：为什么现在平均耗时还是 8 秒左右？
A：主要耗时来自 LLM 审计和外部搜索。这个项目更偏消费审计和解释型建议，不是低延迟接口。优化后 Web Search 已经从平均约 32 秒降到约 2.6 秒，整体从约 51 秒降到约 8.75 秒。后续还可以通过缓存、缩短 prompt、流式输出和更快模型继续优化。

Q9：项目后续怎么扩展？
A：第一，云端持久化迁移到 Supabase 或 PostgreSQL。第二，构建更大的价格评估集，统计估价误差。第三，增加账本编辑和查询能力。第四，把 Trace 做成更完整的可视化看板。第五，如果流程复杂化，可以考虑迁移到 LangGraph。

Q10：这个项目有什么不足？
A：第一，目前还是个人项目，未做多用户权限和高并发部署。第二，云端文件持久化不是最终方案。第三，价格准确性还需要更大样本评估。第四，RAG 效果还没有做系统性的消融实验。第五，当前 Agent 流程适合固定业务链路，还不是通用多 Agent 平台。

二十、如果面试官质疑项目太小
