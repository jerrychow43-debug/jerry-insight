# Lesson 03 - B 树与 B+ 树

## 这节课讲了什么

这一节讲 B 树和 B+ 树。它和上一节红黑树属于同一组主题：都是偏查找的数据结构。

老师解释了为什么数据结构课重点讲红黑树、B 树 / B+ 树、哈希、布隆过滤器，而不是花大量时间讲链表排序。原因是这些结构在工程里经常用于快速查找、索引、缓存和存储系统。

这节课的核心问题是：如果数据不在内存里，而是在磁盘里，为什么普通二叉树不合适？B 树 / B+ 树为什么更适合数据库和文件系统？

## 重点

1. 二叉树层高较高，节点越多，查找时比较和寻址次数越多。
2. 如果节点在内存里，指针寻址成本相对低；如果节点在磁盘里，每访问一个节点都可能触发磁盘 IO。
3. 磁盘 IO 很慢，所以磁盘索引结构要尽量降低树高。
4. B 树 / B+ 树通过多叉节点降低树高。
5. B 树和普通多叉树的关键区别是：B 树约束所有叶子节点在同一层。
6. B+ 树相比 B 树更适合范围查询和数据库索引。

## 为什么需要 B 树

红黑树是二叉树，一个节点最多两个孩子。

如果有 1024 个节点，二叉树大约需要 10 层。查找时可能要沿着树一层层比较和寻址。

如果这些节点都在磁盘页里，每进入下一层都可能需要一次磁盘寻址。磁盘寻址比内存访问慢很多，因此二叉树不适合作为大规模磁盘索引。

B 树的思路是：一个节点里放多个 key，并拥有多个孩子。这样每个节点可以代表一整页数据，树的层高会大幅降低。

例如四叉树比二叉树层高低。如果叉数进一步增大，比如一个节点有很多 key 和孩子，那么同样规模的数据只需要很少几层就能定位。

## B 树的核心性质

课程里的数学定义比较严谨，可以先理解成下面几条：

1. 每个节点最多有 M 棵子树。
2. 有 K 棵子树的节点，通常有 K - 1 个 key。
3. 节点里的 key 是有序的。
4. 除根节点外，每个节点至少有约 M / 2 棵子树。
5. 所有叶子节点在同一层。

最重要的是第 5 条：所有叶子节点在同一层。这保证了树不会退化成长链表。

## B 树和多叉树的区别

普通多叉树只是“一个节点可以有多个孩子”，但如果没有平衡约束，也可能退化成链式结构。

B 树不只是多叉，它还要求节点数量、key 数量、叶子层级满足约束，所以它能保持较低树高。

## B 树和 B+ 树的区别

B 树：

- 所有节点都可以存储真实数据。
- 查找某个 key 时，可能在内部节点就找到数据。

B+ 树：

- 只有叶子节点存储真实数据。
- 内部节点只作为索引。
- 叶子节点通常按顺序串起来，适合范围查询。

## 为什么数据库更常用 B+ 树

B+ 树的内部节点只存索引，不存完整数据，所以一个磁盘页里可以放更多 key。

这会带来两个好处：

1. 叉数更大，树更矮，磁盘 IO 更少。
2. 所有数据都在叶子节点，并且叶子节点有序连接，范围查询更方便。

比如查询：

```sql
where price between 100 and 500
```

B+ 树找到第一个满足条件的叶子节点后，可以顺着叶子链表继续往后扫。

## 代码层面的结构

课程里给了 B 树节点的大致定义思路：

- `keys`：当前节点保存的 key 数组。
- `children`：子节点指针数组。
- `num`：当前节点已有 key 的数量。
- `leaf`：是否是叶子节点。

可以抽象成：

```cpp
struct BTreeNode {
    std::vector<int> keys;
    std::vector<BTreeNode*> children;
    int num;
    bool leaf;
};
```

真正工业实现会更复杂，还要考虑磁盘页、缓存、序列化、并发控制等问题。

## 插入时的分裂

B 树插入的关键是节点满了要分裂。

普通节点分裂：

1. 找到满的子节点。
2. 中间 key 上移到父节点。
3. 左右两部分分裂成两个节点。
4. 父节点增加一个 key 和一个 child。

根节点分裂比较特殊：

1. 根节点满了。
2. 先创建一个新的空根节点。
3. 新根节点的第 0 个 child 指向旧根节点。
4. 再对旧根节点做分裂。
5. 树高增加 1。

## 删除时的合并与借位

删除比插入更复杂。

当某个节点 key 数量不足时，可能需要：

- 向兄弟节点借位。
- 和兄弟节点合并。
- 父节点的 key 下移。

老师强调：删除代码不要求手写，但要理解流程。
