# Lesson 05 - 设计模式导论、观察者模式、策略模式

## 这节课讲了什么

这一节是设计模式第一讲。课程没有按 23 种设计模式逐个讲，而是先讲设计模式的本质、使用前提、面向对象基础、设计原则，然后重点讲了观察者模式和策略模式。

这一节最重要的不是背模式名字，而是学会一个判断方法：

先分析需求里的稳定点和变化点，再决定要不要用设计模式。

## 设计模式是什么

设计模式是软件开发过程中经过验证的、用于解决特定环境下重复出现问题的解决方案。

换成人话：

设计模式是解决某类常见问题的固定套路。

但是它不是万能药。只有当需求里既有稳定点，又有反复出现的变化点时，设计模式才有价值。

## 什么时候不要用设计模式

1. 需求非常稳定，几乎不会变化：
   没必要引入额外抽象。

2. 需求完全不稳定，变化没有方向：
   设计模式也救不了，可能更适合脚本、配置、热更新等方式。

3. 不知道模式解决什么问题，只是为了显得高级：
   容易过度封装，让代码更难维护。

## 设计模式解决什么问题

设计模式的目标是：

在需求变化时，用尽量少的代码修改适应变化。

课程里用了“整洁房间和好动的猫”的比喻：

- 整洁房间：稳定点。
- 好动的猫：变化点。
- 笼子：设计模式。

设计模式不是消灭变化，而是把变化限制在可控范围内。

## 学设计模式需要什么基础

### 面向对象三大特征

1. 封装：
   隐藏实现细节，实现模块化。

2. 继承：
   在不修改原有类的基础上扩展功能。

3. 多态：
   同一个接口在运行时表现出不同实现。

### C++ 多态重点

课程重点讲了动态多态：

- 基类中声明 `virtual` 函数。
- 子类重写虚函数。
- 用基类指针或引用指向子类对象。
- 调用虚函数时，通过虚函数表在运行时决定调用哪个实现。

例如：

```cpp
Base* p = new Derived();
p->run();
```

如果 `run` 是虚函数，调用的是 `Derived::run`。

很多设计模式都依赖 C++ 的动态多态。

## 设计原则

这一节反复强调几个原则：

### 依赖倒置

实现和使用者都依赖抽象，而不是依赖具体类。

也就是：

高层模块不要直接依赖低层实现，二者都依赖接口。

### 接口隔离

客户程序不应该依赖它不需要的接口。

通俗理解：

我只关心你有没有这个能力，不关心你是什么具体类，也不关心你还有什么其他能力。

### 面向接口编程

变量类型尽量声明为接口或抽象基类，而不是具体类。

### 开闭原则

对扩展开放，对修改关闭。

新增功能尽量通过新增类、继承、组合完成，而不是反复修改旧类。

## 观察者模式

### 解决什么问题

观察者模式解决的是一对多依赖。

一个对象发生变化时，多个对象需要收到通知并做出反应。

例如：

- 数据中心数据变化。
- 多个显示终端需要刷新。
- 多个模块监听同一个事件。

### 稳定点

被观察者发布通知的流程是稳定的。

它只需要知道：

有一批对象需要被通知。

### 变化点

观察者的类型和数量会变化。

今天可能是屏幕显示，明天可能是日志模块、告警模块、统计模块。

### 设计思路

定义一个观察者接口，例如：

```cpp
class Observer {
public:
    virtual ~Observer() = default;
    virtual void update() = 0;
};
```

被观察者维护一个观察者列表：

```cpp
std::vector<Observer*> observers;
```

当数据变化时，调用每个观察者的 `update()`。

新增观察者时，只需要：

1. 写一个类继承 `Observer`。
2. 实现 `update()`。
3. 注册到被观察者中。

### 和 Agent 项目的关系

Jerry-Insight Agent 里可以用观察者模式处理事件：

- 记账成功后通知日志模块。
- 余额变化后通知提醒模块。
- 商品价格更新后通知缓存模块。
- Trace 产生后通知分析模块。

## 策略模式

### 解决什么问题

策略模式解决的是一组算法可以互相替换的问题。

课程里的例子是商场促销：

- 春节活动。
- 七夕活动。
- 五一活动。
- 国庆活动。
- 圣诞活动。

如果全部写成 `if else`，新增活动或修改活动时，会不断改同一个促销类。

### 稳定点

客户程序调用促销算法的关系是稳定的。

也就是：

促销活动总是需要调用某个算法来计算结果。

### 变化点

具体促销算法会变化：

- 新增算法。
- 修改算法内容。
- 替换算法。

### 设计思路

定义策略接口：

```cpp
class PromotionStrategy {
public:
    virtual ~PromotionStrategy() = default;
    virtual double calculate(double price) = 0;
};
```

不同活动实现不同策略：

```cpp
class DiscountStrategy : public PromotionStrategy {
public:
    double calculate(double price) override {
        return price * 0.8;
    }
};
```

促销类依赖抽象接口：

```cpp
class Promotion {
public:
    explicit Promotion(PromotionStrategy* strategy)
        : strategy_(strategy) {}

    double checkout(double price) {
        return strategy_->calculate(price);
    }

private:
    PromotionStrategy* strategy_;
};
```

这就是依赖注入：

把具体策略通过构造函数或普通函数传进去。

### 策略模式消除什么

策略模式常用于消除大量平行的 `if else`。

但注意，不是看到 `if else` 就一定要用策略模式。只有当这些分支代表一组可替换算法，并且经常变化时，才适合用。

## 本节课后任务

给 Linux C++ 项目加一个事件系统和策略系统。

### 任务 1：AgentEventBus

实现一个简单观察者模式：

事件类型：

- `ExpenseAdded`
- `PriceUpdated`
- `TraceCreated`

观察者：

- `ConsoleLogger`
- `BudgetNotifier`
- `TraceCollector`

功能：

- 注册观察者。
- 发布事件。
- 所有订阅者收到通知并打印处理结果。

### 任务 2：ExpenseDecisionStrategy

实现一个策略模式：

定义消费决策策略接口：

```cpp
class ExpenseDecisionStrategy {
public:
    virtual ~ExpenseDecisionStrategy() = default;
    virtual std::string decide(double price, double budgetLeft) = 0;
};
```

实现三个策略：

- `StrictStrategy`：严格省钱。
- `BalancedStrategy`：平衡建议。
- `LooseStrategy`：宽松消费。

命令行支持切换策略：

```text
strategy strict
decide 199 850
strategy balanced
decide 199 850
```

## 做到什么程度算过关

能讲清楚：

1. 设计模式不是越多越好。
2. 用设计模式前要先找稳定点和变化点。
3. 观察者模式解决一对多通知。
4. 策略模式解决一组算法互相替换。
5. C++ 里常靠抽象基类、虚函数、基类指针实现多态。
6. 依赖注入可以让类依赖接口，而不是依赖具体实现。

## 和 Jerry-Insight Agent 的关系

Jerry-Insight Agent 未来可以这样设计：

- 用户记账后发布 `ExpenseAdded` 事件。
- 价格抓取后发布 `PriceUpdated` 事件。
- Agent 链路完成后发布 `TraceCreated` 事件。
- 不同模块作为观察者响应事件。

消费建议可以用策略模式：

- 严格模式：优先省钱。
- 平衡模式：兼顾预算和需求。
- 宽松模式：允许一定享受型消费。

这样 Agent 不再把所有逻辑写死在一个大函数里，而是把变化点拆出来。

## 简历表达

在 C++ 消费事件处理服务中引入观察者模式和策略模式：通过事件总线解耦记账、日志、提醒和 Trace 收集模块；通过消费决策策略接口支持严格、平衡、宽松等多种决策算法，提升系统扩展性并减少条件分支耦合。
