# Lesson 04 - 哈希的工程应用

## 这节课讲了什么

这一节讲哈希在后端开发里的常见应用。

它不是让你研究哈希算法内部的数学实现，而是从工程角度理解哈希能解决什么问题、怎么选择哈希函数、散列表怎么工作、冲突怎么处理，以及布隆过滤器和一致性哈希为什么重要。

这一节和前面的红黑树、B 树 / B+ 树放在一起看，都是“强查找”相关的数据结构。

## 重点

1. 红黑树、B 树 / B+ 树通过“比较 + 有序结构”提升查找效率。
2. 散列表通过“key 到存储位置的映射”提升查找效率。
3. 散列表通常可以做到平均 O(1) 查询，但牺牲了有序性，并且用空间换时间。
4. 哈希函数要计算快，并且分布均匀。
5. 哈希冲突无法完全避免，要靠冲突处理和扩容缩容控制。
6. 布隆过滤器可以快速判断“某个元素一定不存在 / 可能存在”。
7. 一致性哈希用于分布式缓存、Redis 集群、负载分流等场景。

## 哈希解决的问题

课程举了几个典型例子：

- Word 判断单词是否拼写正确。
- 爬虫判断某个 URL 是否已经爬过。
- 垃圾邮件过滤。
- 判断某个人是否在黑名单、网逃名单中。
- 大文件查重或分流。
- 单台机器处理不了时，把数据分到多台机器。

这些问题的共同点是：

需要在大量数据中快速判断某个 key 是否存在。

## 散列表和平衡树的区别

平衡树：

- 通过比较 key 保持有序。
- 查询复杂度是 O(log n)。
- 支持有序遍历和范围查询。

散列表：

- 不靠比较保持有序。
- 通过哈希函数计算 key 应该存储的位置。
- 平均查询复杂度是 O(1)。
- 不天然支持有序遍历和范围查询。
- 空间占用通常更高。

一句话：

平衡树是“有序换查找效率”，散列表是“映射关系换查找效率”。

## 哈希表的组成

散列表主要由两部分组成：

1. 哈希函数。
2. 存储数组。

插入流程：

1. 对 key 进行哈希，得到 32 位或 64 位整数。
2. 对数组长度取模，得到槽位下标。
3. 把 key-value 节点放到对应槽位。

查询流程：

1. 对同一个 key 再次哈希。
2. 对相同数组长度取模。
3. 定位到同一个槽位。
4. 在槽位中查找目标节点。

## 哈希函数怎么选

好的哈希函数要满足：

1. 计算速度快。
2. 随机分布性强。
3. 对相近字符串也能打散。

课程里提到：

- MurmurHash2：工程里常用，速度和质量比较均衡。
- SipHash：Redis 使用，适合处理相近字符串 key。
- CityHash：使用广泛，分布性较好。

这一节不用研究哈希函数内部数学细节，只要知道如何从工程角度选择。

## 哈希冲突

哈希冲突是指多个 key 经过哈希和取模后落到同一个数组槽位。

冲突一定会发生，原因可以用抽屉原理理解：

如果槽位数量有限，而元素数量越来越多，必然有多个元素落到同一个槽位。

## 负载因子

负载因子用于描述哈希表的拥挤程度：

```text
负载因子 = 已存储元素数量 / 数组长度
```

负载因子越大，冲突概率越高。

当负载因子超过阈值时，需要扩容并重新哈希。

## 冲突处理方式

### 链表法 / 拉链法

数组的每个槽位指向一个链表。

如果多个 key 落到同一个槽位，就把它们挂到链表上。

常见应用：

- Redis
- C++ `std::unordered_map`
- Java HashMap 的早期结构

优点：

- 实现简单。
- 使用广泛。

缺点：

- 冲突严重时，链表变长，查询退化。

### 开放寻址法

所有元素都直接存储在数组里。

如果当前位置冲突，就按某种探测算法找下一个可用位置。

常见探测方式：

- 线性探测。
- 二次探测。
- 双重哈希。

布隆过滤器里的位图思想和开放寻址有一定关联，但布隆过滤器本身不是普通哈希表。

## 扩容和缩容

当元素越来越多，负载因子过高时，需要扩容。

扩容通常会：

1. 创建更大的数组。
2. 遍历旧数据。
3. 重新计算每个 key 的新位置。
4. 插入到新数组。

删除很多元素后，也可能触发缩容。

## STL unordered_map 要理解什么

重点不是背源码，而是理解：

1. 它是哈希表。
2. 查询平均 O(1)。
3. 不保证有序。
4. 有 bucket 概念。
5. 冲突通常通过链表或类似结构处理。
6. 负载因子过高会 rehash。

## 布隆过滤器

布隆过滤器用于快速判断一个元素是否存在。

它的结论有两个：

1. 判断不存在：一定不存在。
2. 判断存在：可能存在，有误判概率。

布隆过滤器的特点：

- 空间占用很低。
- 查询速度快。
- 有假阳性。
- 不能直接删除普通元素。

典型应用：

- MySQL 缓存方案中防止缓存穿透。
- RocksDB 中判断 key 是否可能在某个 SSTable 中。
- 爬虫 URL 去重。
- 黑名单预过滤。

布隆过滤器的参数：

- `n`：预计插入元素数量。
- `p`：可接受误判率。

根据 `n` 和 `p` 可以计算：

- 位图大小。
- 哈希函数数量。

## 一致性哈希

一致性哈希用于分布式系统中的数据分布。

普通取模的问题：

```text
server = hash(key) % server_count
```

如果机器数量变化，比如从 3 台变 4 台，几乎所有 key 的映射都会改变，导致大面积缓存失效。

一致性哈希的思路：

1. 把哈希空间看成一个环。
2. 把服务器节点哈希到环上。
3. 把 key 也哈希到环上。
4. key 顺时针找到的第一个服务器，就是它的归属节点。

这样新增或删除节点时，只会影响局部数据。

## 哈希迁移和哈希偏移

哈希迁移：

新增节点后，部分 key 原本属于旧节点，现在应该属于新节点，需要把这一段数据迁移过去。

哈希偏移：

如果真实节点数量太少，节点在环上分布不均匀，就会导致某些机器数据特别多，某些机器数据特别少。

解决方式：

增加虚拟节点。

## 虚拟节点

一个真实物理节点可以映射成多个虚拟节点。

例如：

```text
192.168.1.100:6000#1
192.168.1.100:6000#2
192.168.1.100:6000#3
...
```

每个虚拟节点都哈希到环上。

好处：

1. 增加样本数，让分布更均匀。
2. 减少单次节点变更时的数据迁移量。
3. 提升分布式缓存的稳定性。

实现时可以用：

```cpp
std::map<uint32_t, std::string> ring;
```

`std::map` 是有序结构，可以用 `upper_bound` 找到顺时针方向第一个大于当前 key 哈希值的虚拟节点。

## 大数据面试套路

课程最后提到两个很重要的面试思路：

1. 大文件处理：
   用哈希把大文件拆成多个小文件。

2. 单台机器处理不过来：
   用哈希把数据分流到多台机器。

核心原因：

- 相同 key 会被分到同一个位置。
- 哈希有强随机分布性，可以让数据尽量均匀分散。

## 本节课后任务

做一个 C++ 哈希工具模块，分两步。

### 任务 1：实现简单 HashTable

用链表法实现一个简化版哈希表：

支持：

- `put key value`
- `get key`
- `remove key`
- `stats`
- `exit`

`stats` 输出：

- bucket 数量。
- 元素数量。
- 负载因子。
- 最大链表长度。

这样可以直观看到冲突和负载因子的关系。

### 任务 2：实现一致性哈希 Demo

用 `std::map<uint32_t, std::string>` 实现一致性哈希环。

支持：

- 添加物理节点。
- 给每个物理节点添加多个虚拟节点。
- 输入 key，返回它落到哪台机器。
- 新增节点后观察哪些 key 需要迁移。
